// the project

An agentic AI that knows me

I built a personal AI assistant that answers questions about me by orchestrating five specialist sub-agents — each reading from a different data source — and synthesising the results into a single coherent answer.

💡

Key insight: The orchestrator LLM never answers from its own training knowledge. It is always forced to call tools first, get real data from my actual databases and documents, then synthesise a grounded response. This is the core principle of agentic AI: action before answer.

1
User asks a question
A natural-language question arrives via the Gradio UI, a custom web interface, or the FastAPI WebSocket endpoint. No special syntax required.
2
Orchestrator reads tool schemas & decides
Claude reads the descriptions of all five tools and reasons about which agent(s) can answer the question. For multi-part questions it picks several at once.
3
Parallel tool execution
Selected agents run simultaneously via ThreadPoolExecutor — a SQLite query and a vector store search happen at the same time, not one after another.
4
Results injected back into context
Tool results are appended to the conversation as structured messages. The LLM reads all of them together and decides if it needs more data — or is ready to answer.
5
Grounded, streamed final answer
Claude synthesises a concise answer from real data and streams it token-by-token to the user. Every answer cites which source(s) it used.

// under the hood

Real code, real patterns

Every agentic concept is implemented explicitly — you can read and understand exactly what's happening at every step.

Parallel tool execution

When the orchestrator decides to call multiple agents simultaneously, they run in parallel threads — not sequentially. A SQLite query and a vector search finish together, not one after another.

orchestrator.py 
with ThreadPoolExecutor(max_workers=len(calls)) as executor:

    # submit all agents simultaneously
    futures = {
        executor.submit(self._dispatch, name, args): call_id
        for call_id, name, args in calls
    }

    # process results as they finish
    for future in as_completed(futures):
        call_id = futures[future]
        results[call_id] = future.result()

RAG pipeline

The knowledge base agent embeds the question, finds semantically similar chunks from the vector store, then generates a grounded answer — not a hallucination.

knowledge_base_agent.py 
def run(self, question: str) -> str:
    # 1. embed question → same space as documents
    query_vector = self._embed_query(question)

    # 2. cosine similarity search → top 5 chunks
    results = self._collection.query(
        query_embeddings=[query_vector],
        n_results=5,
    )

    # 3. Claude answers from chunks only
    return self._generate(question, chunks)

Streaming responses

The final answer streams token-by-token via an async generator. The UI also receives status events during tool execution — "Searching knowledge base…" — so users know what's happening.

orchestrator.py 
async def run_stream(self, message, history):
    async with async_client.messages.stream(
        model=ORCHESTRATOR_MODEL,
        messages=messages,
    ) as stream:
        async for text in stream.text_stream:
            yield {"type": "token", "text": text}
    yield {"type": "done"}

// what i understand

Agentic AI concepts, hands-on

Building without frameworks means understanding each pattern deeply enough to implement it from scratch.

01
Multi-agent orchestration
A central LLM routes questions to specialist agents based on tool descriptions alone. No rules, no if-statements — the model reasons about which tool fits and calls it.
02
Tool-calling loops
The orchestrator loops: call LLM → get tool requests → run tools → inject results → call LLM again. The loop continues until stop_reason is "end_turn". Implemented for both Anthropic and OpenAI formats.
03
RAG (Retrieval-Augmented Generation)
Documents are chunked, embedded into vectors, and stored in ChromaDB. At query time the question is embedded and compared via cosine similarity. The top-K chunks ground the LLM's answer in real content.
04
Parallel agent execution
ThreadPoolExecutor runs multiple agents simultaneously. as_completed() processes results as they arrive. A 1s SQLite call and a 3s web scrape finish at second 3, not second 4.
05
Async streaming
The final answer is streamed token-by-token via AsyncAnthropic. Tool-call rounds block on the executor (non-blocking for the event loop). Status events keep the UI responsive throughout.
06
Provider-agnostic design
Anthropic, OpenAI, and Ollama are abstracted behind a single config flag. Tool schemas are maintained in Anthropic format and auto-converted to OpenAI format — one source of truth.
07
Conversation memory
A ConversationMemory class converts Gradio's chat history format into the API's expected message list on each request, maintaining context across a multi-turn conversation.
08
Resilience & retry logic
API overload (529) errors are caught and retried with exponential backoff. Individual agent failures are isolated — the orchestrator continues with results from the agents that succeeded.
